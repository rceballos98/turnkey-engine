# Turnkey Engine

FastAPI microservice with a background worker and Postgres job queue. The agent pipeline (Firecrawl + Socrata + Claude) is a placeholder — this is the scaffolding.

## Architecture

```
Client → turnkey-engine-api (FastAPI/Gunicorn) → Postgres (Render)
                                                      ↑
         turnkey-engine-worker (polling loop) ─────────┘
```

- **API** — FastAPI app behind Gunicorn. Handles auth, creates reports, enqueues jobs.
- **Worker** — Background process that polls `job_queue`, claims jobs via `SELECT ... FOR UPDATE SKIP LOCKED`, runs the agent, and writes results back.
- **Postgres** — Render free-tier Postgres. Tables managed by Alembic migrations.

## Local Development

```bash
# 1. Create virtualenv and install deps
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 2. Start Postgres
docker run -d --name turnkey-pg -e POSTGRES_DB=turnkey -e POSTGRES_PASSWORD=postgres -p 5432:5432 postgres:16

# 3. Configure env
cp .env.example .env

# 4. Run migrations
alembic upgrade head

# 5. Start API
uvicorn app.main:app --reload

# 6. Start worker (separate terminal)
python -m app.worker
```

## Testing

Integration tests run against the real local Postgres — no mocking.

```bash
source .venv/bin/activate
pytest tests/ -v
```

Tests cover: health check, auth (missing/invalid), report creation, 404 handling, and the full lifecycle (create → enqueue → claim → process → retrieve).

## API

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/health` | No | Health check (pings DB) |
| `POST` | `/reports` | Bearer token | Create a report and enqueue for processing |
| `GET` | `/reports/{id}` | Bearer token | Get report by ID |

Auth: `Authorization: Bearer <INTERNAL_API_KEY>`

### Example

```bash
# Create a report
curl -X POST https://turnkey-engine-api.onrender.com/reports \
  -H "Authorization: Bearer <INTERNAL_API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"query": "What are the top 3 industries in Oregon?"}'

# Check report status (use the ID from the response above)
curl https://turnkey-engine-api.onrender.com/reports/<REPORT_ID> \
  -H "Authorization: Bearer <INTERNAL_API_KEY>"
```

## Deploy to Render

Deployed via Render Blueprint (`render.yaml`). Auto-deploys on push to `main`.

| Service | Type | Plan |
|---------|------|------|
| `turnkey-engine-api` | Web Service | Starter |
| `turnkey-engine-worker` | Background Worker | Starter |
| `turnkey-db` | Postgres 16 | Free |

### Environment Variables

| Variable | Where to get it |
|----------|----------------|
| `DATABASE_URL` | Render Postgres → Internal Database URL |
| `INTERNAL_API_KEY` | Auto-generated by Render |
| `ANTHROPIC_API_KEY` | [Anthropic Console](https://console.anthropic.com/) |
| `FIRECRAWL_API_KEY` | [Firecrawl Dashboard](https://firecrawl.dev/) |
| `BASE_URL` | Your `.onrender.com` URL |

### First-time setup

1. Push repo to GitHub
2. Create a **Blueprint** in Render, point it at the repo
3. Create a **Postgres** instance in Render (free tier, same region)
4. Set `DATABASE_URL` on both API and worker to the **Internal Database URL**
5. Render runs `alembic upgrade head` as a pre-deploy step to create tables
6. `INTERNAL_API_KEY` is auto-generated
